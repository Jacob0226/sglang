AMD MI355X — single-stream (winner)

Stream 8 (single)  |  ≈ 285 μs / layer  |  8k1k conc4 median TPOT 20.48 ms (faster than PR #23562 baseline 21.21 ms). CatArrayBatchedCopy eliminated via q_cat reuse.

⚠ Why the timeline below is single-stream
The A_v4 dual-stream layout (overlap NSA indexer on alt with [q_b_proj + bmm w_kc + fused_qk_rope_cat] on cur) regressed ~30 μs / layer on MI355X — measured by re-running the same code with alt_stream forced to None (MyBranch_NoDualStream).

Measured kernel-duration deltas (single → dual-stream):
  • Memory-bound indexer kernels (Hadamard, act_quant, paged_mqa, topk_transform): each +0.5–2.4 μs, +8 μs total / layer.
  • Compute-bound kernels (main_kernel, bmm_w_vc, etc.): each +0.5–1.5 μs, +5 μs total / layer.
  • AllReduce: aiter::cross_device_reduce_1stage<bf16,8> jumps from 9.5 μs (single) to 23 μs (dual), same kernel, same 8-rank topology, +13 μs × 2 ARs = +26 μs / layer.
Hypothesised causes (consistent with the data but not yet verified with rocprof HW counters):
  • HBM (8 TB/s) channel / queue contention between concurrent kernels.
  • Per-stream peer-fence in aiter's cross_device_reduce extends when alt has outstanding writes.
A_v4's theoretical saving (gap-fill ∥ indexer ≈ −10 μs) is dwarfed by the cost. The dual-stream code is preserved behind SGLANG_ENABLE_HIP_DUAL_STREAM=1 for future ROCm releases that may fix this.
