★ ROCm 7.13 + 4-stream: bubble unchanged, but kernels are slower
Surprising finding from the trace below: real-stall bubble is ~the same as ROCm 7.2 dual (16.3 vs 16.5 µs / layer) — the HIP-graph fence overhead did not get worse with more streams. The +25–54 µs / layer regression comes from kernels themselves running slower:
• AR per call: baseline 10.3 → ROCm 7.2 dual 18.2 → ROCm 7.13 dual 22.3 µs at this layer (median over the trace 25.7 µs). +4–7 µs / call × 2 calls = +8–15 µs / layer.
• Indexer chain: 70 → 91 µs / layer (+21 µs). Memory-bound kernels (Hadamard, act_quant, paged_mqa, topk) each pay 0.5–2 µs more vs ROCm 7.2 dual; HBM contention with 4 concurrent streams is worse than 2.
• GEMMs (q_b_proj, w_kc, ck_gemm): aiter rebuilt with PREBUILD_KERNELS=1 but no tuned configs (/tmp/aiter_configs/*.csv empty), so first-touch shapes hit "torch solution:0" / "skinny solution:2" defaults. +9 µs in 9 GEMMs mostly from this.
Net: dual-stream regression went from +49 → +66–95 µs / layer with the ROCm 7.13 stack. The 4-stream layout doesn't make the layout worse per se; the new user-space stack just hasn't had its tuned configs and warm-up complete.

⚠ Sub-1 µs caveat — profiler accounting, not GPU behavior

The naive reading "B200 has 23 µs / layer of launch micro-gaps, MI355X has 0 — MI355X packs kernels tighter" is wrong. The sub-1 µs column is a CUPTI vs roctracer accounting difference, not a real GPU-utilization gap.

The smoking gun: B200 has 4,296 negative gaps, smallest −6.66 µs. A negative gap (next-kernel start_ts before prev-kernel end_ts) is physically impossible if both ts come from the same event class. The only explanation: CUPTI ts uses different events for start vs end, while roctracer uses coherent ones:

start_tsend_ts
CUDA / CUPTIfirst warp begins on SMlast warp completes on SM
ROCm / roctracerAQL dispatch packet processed (CP layer)kernel completion signal

CP→SM dispatch latency is ~200–500 ns on both platforms. CUPTI exposes it as inter-kernel gap; roctracer absorbs it into the kernel's reported duration. So MI355X's "0 µs gap" doesn't mean 0 GPU idle — it means roctracer charged dispatch latency to the kernel itself. The actual physical dispatch latency is similar on both platforms. Sub-1 µs gaps are not actionable on either side without kernel fusion.

What does matter: MI355X dual-stream's 3 ≥2 µs stalls. All ~5–6 µs each, all caused by HIP-graph dual-stream replay:

1small fence32.6 → 34.2 µs1.6 µsbetween attention-block kernels

Why these are stalls and not dispatch latency: each is 5–6 µs (roughly 10× normal dispatch latency), they don't appear in MI355X single-stream (same kernels, same bookkeeping), and they don't appear in B200 dual-stream either. They are HIP-graph-specific cross-stream wait_event fences. Same root cause class as the AR slowdown — both surface only under dual-stream HIP-graph capture, both should be re-checked on each new ROCm release.

03
Mechanism · primary vs secondary

Two cost contributors that are pure runtime artifacts.

★ primary · runtime artifact

Bubble + AR slowdown

+34.2 µs / layer

Bubble +18.2 µs: GPU sits idle because HIP graph capture inserts cross-stream fences during replay. Same trace, same model, same kernels — the GPU is just waiting.

AR +16.0 µs: aiter::cross_device_reduce_1stage is identical bytes/aiter-build/topology, but mean +7.98 µs / call slower under dual-stream HIP-graph replay (likely peer-fence drains the alt-stream KV-cache writes). Distribution shifts uniformly right.

These together = ~70% of the regression. Both are HIP-runtime / graph-capture artifacts, not properties of the dual-stream attention layout. A future ROCm release that fixes the graph fence cost would recover most of it.

secondary · model-config side-effects

MoE unfusing + indexer contention

+15.8 µs / layer

MoE +8.8 µs: --dual-stream-rocm implies --disable-shared-experts-fusion, which forces aiter to split the shared-expert pipeline into 4 kernels (ck_gemm_preshuffle gate_up + silu + ck_gemm_preshuffle down + add). The dual-stream layout overlaps most of that with the routed-expert chain on cur stream — kernel-duration sum is +46 µs but only +8.8 µs surfaces at wall-clock.

Indexer +7.0 µs: 9 NSA memory-bound kernels each pay 0.7–1.0 µs in dual-stream when the indexer chain on alt competes with cur-stream q_b_proj / kv_b_proj for HBM bandwidth (8 TB/s on MI355X, shared). Compute-bound kernels (topk_transform, cached_indirect) are unaffected.

These would recover if aiter ships a fused shared-expert kernel compatible with the dual-stream MoE path.
