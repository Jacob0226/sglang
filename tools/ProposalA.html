MI355X Decode Layer — Two Rules, One Fast Layout

GLM5-FP8 TP=8 M=1 — measurements from tools/glm5_proposalA_v3_test.py on MI355X (container jacchang_GLM_MultiStream)

Two independent HIP-graph rules both matter — get them both right and you get the Scenario 2 layout AND ~19 μs of speedup.

Rule 1 — dispatch-order picks the physical stream. At each fork, the branch whose first kernel is captured first keeps the current physical stream (phys 0). The later-captured branch gets a fresh aux physical stream. HIP's replay scheduler ignores which cuda.Stream object you dispatched to.

Rule 2 — wait_stream position decides when alt starts. alt.wait_stream(cur) records an event at cur's current state. Place it AFTER a heavy cur kernel that alt doesn't actually need (e.g. q_b_proj), and alt stalls for the duration of that kernel even though no real data dependency exists.

Combined fix: (1) move alt_stream.wait_stream(cur) BEFORE phase2_current so alt only waits for phase1, and (2) keep the cur-side gap-fill dispatched before the with stream(alt): block so cur stays on phys 0 and indexer goes to the aux stream. This is Proposal A v4 — the fastest variant and the one with the clean Scenario 2 layout.

The two rules in code

# Baseline — wait_stream early, cur dispatched first → works, but no gap-fill
alt_stream.wait_stream(cur)                    #  Rule 2 ✓  alt waits for phase1 only
q, q_lora = phase2_current(...)                #  Rule 1 ✓  1st dispatch → cur stays on phys 0
with torch.cuda.stream(alt_stream):
    topk_ids, kv_normed = phase2_alt(...)      #  alt runs in parallel starting at phase1 end
cur.wait_stream(alt_stream)

# A_v2 — alt dispatched first. Parallelism OK, but labels flip on the trace.
q, q_lora = phase2_current(...)
alt_stream.wait_stream(cur)                    #  Rule 2 ✗  after q_b_proj → alt starts late
with torch.cuda.stream(alt_stream):
    topk_ids = phase2_alt_proposalA(...)       #  Rule 1 ✗  1st dispatch → phys 0 (wanted phys 4)
kv_normed = aiter_rms_norm(...)                #                2nd dispatch → phys 4 (wanted phys 0)
q_all = phase3_pre_wait(...)
cur.wait_stream(alt_stream)

# A_v3 — cur dispatched first, wait_stream still late → clean layout, alt still late
q, q_lora = phase2_current(...)
alt_stream.wait_stream(cur)                    #  Rule 2 ✗  still after q_b_proj → alt t=71 μs (20 late)
kv_normed = aiter_rms_norm(...)                #  Rule 1 ✓  1st → phys 0
q_all = phase3_pre_wait(...)
with torch.cuda.stream(alt_stream):
    topk_ids = phase2_alt_proposalA(...)       #  Rule 1 ✓  2nd → phys 4
cur.wait_stream(alt_stream)

# A_v4 — BOTH rules satisfied → fastest AND cleanest
alt_stream.wait_stream(cur)                    #  Rule 2 ✓  before q_b_proj → alt starts at t=50 μs
q, q_lora = phase2_current(...)                #  Rule 1 ✓  1st cur dispatch → cur on phys 0
kv_normed = aiter_rms_norm(...)
q_all = phase3_pre_wait(...)
with torch.cuda.stream(alt_stream):
    topk_ids = phase2_alt_proposalA(...)       #  Rule 1 ✓  dispatched last → phys 4
cur.wait_stream(alt_stream)
Why the wait_stream can safely move before phase2_current: phase2_current returns (q, q_lat) where the second element is q_lora. The indexer's first kernel is wq_b = fp8_gemm(q_lora, …) — it only consumes q_lat (phase1 output), not the GEMM output q. So alt has no true data dependency on q_b_proj and can start the moment phase1 completes.

HIP kept the indexer on phys 0 because its first kernel was captured before any of the cur-side gap-fill kernels (Rule 1). The parallelism is still correct — you get ~10 μs of speedup — but if you read this trace at face value, it looks like "W_kc / RoPE / Cat are on alt", which is the opposite of what the Python code says. Why we can't do much better than ~15 μs on M=1 decode

The NSA indexer is ~132 μs and is the critical path. On the cur side, the kernels that don't depend on topk_ids total ~18 μs (kv_a_norm 2 + W_kc 6 + RoPE_mla 5 + Cat 5). Proposal A v2 captures 14.5 μs of that 18 μs (≈80%); v3 captures 9.1 μs (≈50%) because of the extra cross-stream wait for attn. To break past ~15 μs we would need to shrink the indexer itself (e.g. smaller wk, fewer hadamards), or raise M so phase1 / q_b_proj grow enough to absorb more idle.

Proposal C (production-style K/Q indexer split) and Proposal D (Proposal A + wq_b on cur) both made it worse by adding more cross-stream dependency nodes — confirmed in the previous iteration of glm5_proposalC_test.py.
